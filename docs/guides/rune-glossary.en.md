# Rune Glossary

Short glossary to make Rune docs easier to read.
Some terms are intentionally kept in English for technical clarity.

## Core Terms

| Term | Plain meaning |
|------|---------------|
| `plan` | Implementation plan file (markdown under `plans/`) |
| `workflow` | Sequence of steps for a task (for example `devise`, `strive`, `appraise`) |
| `phase` | Sub-step inside a larger workflow (especially `arc`) |
| `gate` / `quality gate` | Quality checkpoint that can block progress on failure |
| `artifact` | Output file generated by a workflow |
| `checkpoint` | Saved intermediate state used by `--resume` |
| `resume` | Continue from checkpoint instead of starting over |
| `Ash` | Specialized Rune agent |
| `Roundtable Circle` | Parallel multi-reviewer coordination model |
| `TOME` | Consolidated findings report from review/audit |
| `finding` | Issue detected by reviewers (typically P1/P2/P3) |
| `mend` | Workflow that fixes findings |
| `ward command` | Quality command such as lint/test/typecheck |
| `dedup` | Remove duplicate findings across agents |
| `Echoes` | Persistent Rune memory for project patterns/decisions |
| `Remembrance` | Human-facing docs promoted from Echoes |
| `summon` | Activate an agent/team for a specific workflow or phase |
| `Forge Gaze` | Topic-aware agent matching for plan enrichment |
| `blast radius` | Impact scope of a change |
| `dry-run` | Preview execution scope without running agents |
| `worktree` | Isolated repository copy for parallel work and lower conflict risk |

## Finding Priority

| Level | Meaning |
|-------|---------|
| `P1` | Critical, fix immediately |
| `P2` | High priority, fix soon |
| `P3` | Medium/low priority, can be scheduled later |

## Minimum Path for New Users

1. `/rune:plan`
2. `/rune:work`
3. `/rune:review`
4. If TOME exists: `/rune:mend tmp/reviews/{id}/TOME.md`
