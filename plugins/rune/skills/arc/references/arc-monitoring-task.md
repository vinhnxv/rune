# Arc Monitoring Task — Reference Document

This document describes the monitoring prompt logic used by the arc scheduler. The monitoring task is created via `CronCreate` when an arc starts and automatically detects and resumes arcs that stop unexpectedly.

## Monitoring Prompt Template

Generated by `buildArcMonitoringPrompt(checkpoint)` in `arc-checkpoint-init.md`:

```text
[ARC-SCHEDULER MONITORING] Arc ID: {checkpoint.id}

You are a monitoring task for an active Rune arc pipeline. Check if the arc needs to be resumed.

1. Read the file .claude/arc-phase-loop.local.md
2. Parse its YAML frontmatter

Decision tree:
- If file does not exist → arc was cleaned up. Delete this monitoring task: CronDelete task_id={cron_task_id}
- If user_cancelled=true or stop_reason="completed" or stop_reason="user_cancel" → arc finished or was cancelled. Delete this task: CronDelete task_id={cron_task_id}
- If active=true AND no phase is currently in_progress in the checkpoint at .claude/arc/{checkpoint.id}/checkpoint.json → unexpected stop detected. Check cooldown first (see below), then Run: /rune:arc --resume
- If active=true AND a phase IS in_progress → arc is still running normally. No action needed.

Cooldown check (prevents concurrent resumes):
- Read checkpoint.resume_tracking.last_resume_at
- If last_resume_at is within the last {cooldown_minutes} minutes → skip this cycle. Another resume may still be in progress.

Resume limits (read from checkpoint resume_tracking):
- If total_resume_count >= {max_total_resumes} → do NOT resume. Delete this task.
- If consecutive_failures >= {max_consecutive_failures} → do NOT resume. Delete this task.

Before resuming, update checkpoint.resume_tracking: increment total_resume_count, set last_resume_at to now, add entry to resume_history with timestamp and trigger="scheduled_monitor".
After a SUCCESSFUL resume completes a phase: reset consecutive_failures to 0.
```

## Key Concepts

### Why This Is Needed

When a Rune arc pipeline's stop hook fails (due to timeout, error, or other issues), the arc stops mid-execution with no automatic recovery mechanism. The monitoring task provides a best-effort safety net that:

1. Detects unexpected stops (active arc with no in_progress phase)
2. Attempts automatic resume via `/rune:arc --resume`
3. Respects user cancellation (user_cancelled flag)
4. Prevents infinite loops via resume limits

### Architectural Reality

This feature is specifically a **stop-hook-failure recovery mechanism**, NOT a general health monitor. The arc stop hook (`arc-phase-stop-hook.sh`) chains phases by outputting `{"decision":"block"}` on every Stop event — meaning Claude is NEVER idle during normal arc execution. The monitoring task can only fire between turns, so it will never fire during normal operation. It only activates when the stop hook itself fails.

### Key Limitations

1. **Session-scoped**: Cannot recover from complete session crashes (scheduled tasks are session-scoped)
2. **Between-turns only**: If Claude is stuck mid-response, monitoring cannot fire
3. **No catch-up**: If a scheduled time passes while Claude is busy, it fires once when idle
4. **Requires Claude Code >= v2.1.71**: CronCreate/CronDelete/CronList tools
5. **Disabled by `CLAUDE_CODE_DISABLE_CRON=1`**: Environment variable disables all scheduling

## Configuration

Controlled via `talisman.yml` under `arc.scheduler`:

| Setting | Default | Description |
|---------|---------|-------------|
| `enabled` | `true` | Enable/disable scheduled monitoring |
| `interval_minutes` | `15` | Check interval (5, 10, 15, 30) |
| `auto_renew` | `true` | Auto-renew tasks before 3-day expiry |
| `renewal_threshold_hours` | `12` | Renew when < N hours remain |
| `cleanup_orphans_on_start` | `true` | Clean orphaned tasks on session start |
| `auto_resume.enabled` | `true` | Enable automatic resume on unexpected stop |
| `auto_resume.max_total_resumes` | `10` | Maximum total resume attempts per arc |
| `auto_resume.max_consecutive_failures` | `3` | Stop if this many consecutive failures |
| `auto_resume.cooldown_minutes` | `5` | Wait between resume attempts |
| `auto_resume.notify_on_resume` | `true` | Notify user when auto-resume occurs |

## Resume Tracking

The `resume_tracking` object in the checkpoint tracks:

```javascript
resume_tracking: {
  total_resume_count: 0,      // Total resume attempts across arc lifetime
  resume_history: [],          // Array of { timestamp, trigger, phase }
  last_resume_at: null,        // ISO timestamp of last resume attempt
  consecutive_failures: 0      // Failures since last successful phase completion
}
```

### When to Reset `consecutive_failures`

After a **successful phase completion** following a resume, reset `consecutive_failures` to 0. This distinguishes between:
- Transient failures (network, timeout) → reset on success
- Systemic failures (corrupt checkpoint, missing files) → accumulate

## Security Considerations

1. **Prompt injection risk**: The monitoring prompt is injected as plain text into CronCreate. If a malicious actor can modify the checkpoint file, they could alter prompt template fields.
   - Mitigation: Hardcode the state file path, validate `cron_task_id` format

2. **Session isolation**: The monitoring task checks `config_dir` and `session_id` to ensure it only acts on arcs from the same session.

3. **Tool restrictions**: The monitoring prompt is limited to reading files and invoking `/rune:arc --resume` or `CronDelete`.

## Interaction with Cancel Commands

When `/rune:cancel-arc` is invoked:

1. Sets `user_cancelled=true` in state file and checkpoint
2. Sets `stop_reason="user_cancel"`
3. Deletes the monitoring task via `CronDelete`

The monitoring task checks these flags on every fire and self-deletes if the arc was cancelled or completed.

## Task Renewal (3-Day Expiry)

Scheduled tasks expire after 3 days (Claude Code limitation). For long-running arcs:

1. Before expiry, check `scheduler.expires_at`
2. If < `renewal_threshold_hours` remain, create a new task
3. Delete the old task
4. Update `checkpoint.scheduler` with new task info

## Error Handling

| Error | Recovery |
|-------|----------|
| CronCreate fails | Arc continues without monitoring (non-fatal) |
| CronDelete fails | Log warning, continue (task may already be expired) |
| Checkpoint read fails | Skip this cycle, try again next interval |
| Resume fails | Increment `consecutive_failures`, check limits |
| State file missing | Self-delete via CronDelete (arc was cleaned up) |
