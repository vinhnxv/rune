# Arc Monitoring Task — Reference Document

This document describes the dual-layer arc recovery architecture. Layer 1 uses `CronCreate`
for in-session stop-hook failure recovery. Layer 2 uses `SessionStart` hook detection for
cross-session crash recovery.

## Heartbeat Integration (v1.146.0)

The arc heartbeat system tracks activity during arc phases to enable stuck detection:

- **Heartbeat writer**: `scripts/arc-heartbeat-writer.sh` (PostToolUse hook)
- **Tracked tools**: Read, Write, Edit, Bash, Glob, Grep
- **Throttle**: 30-second minimum interval between writes
- **Location**: `tmp/arc/{arc_id}/heartbeat.json` with `last_activity` timestamp
- **Layer 1 use**: CronCreate monitoring displays `last_activity` in stuck arc reports
- **Layer 2 use**: SessionStart hygiene reports `last_activity` for resumable checkpoints

### Heartbeat Format

```json
{
  "arc_id": "arc-abc123",
  "last_activity": "2026-03-10T15:30:45Z",
  "phase": "work"
}
```

### Stuck Detection Heuristics

A phase is considered potentially stuck when:
1. Arc is active (not completed/cancelled)
2. A phase has status "in_progress"
3. `last_activity` is > 10 minutes old
4. No recent Claude responses (session idle)

The monitoring prompt uses `last_activity` to distinguish between:
- **Active arc**: Recent activity, Claude is working → no action needed
- **Stuck arc**: No recent activity, phase stalled → trigger recovery

## Dual-Layer Recovery Architecture (v1.145.0)

### Layer 1 — CronCreate (In-Session, Stop-Hook Failure)

The monitoring task is created via `CronCreate` when an arc starts. It fires when the
stop hook crashes but the session is still alive. During normal arc execution, the stop
hook blocks exit (`exit 2`), keeping Claude busy — so the monitoring task can **only fire
when the stop hook itself fails** (exit non-0/non-2), making Claude idle.

**When it helps**: Stop hook crashes, timeouts, errors — session is alive but the phase
loop stalled.

**When it CANNOT help**: Session crashes (OOM, SIGKILL), terminal closure, machine reboot.
CronCreate tasks are session-scoped and die with the session.

### Layer 2 — SessionStart Hook (Cross-Session, Crash Recovery)

When the session itself crashes, the checkpoint file persists on disk at
`.claude/arc/{id}/checkpoint.json`. On the next session start, `session-team-hygiene.sh`
detects orphaned checkpoints (dead `owner_pid`, not cancelled/completed) and advises the
user to resume via `/rune:arc --resume`.

**When it helps**: Session crashes, OOM kills, terminal closure, machine reboot — any
scenario where the checkpoint survives but the session doesn't.

**Implementation**: See `session-team-hygiene.sh` "Layer 2: Resumable arc detection" block.

### Recovery Matrix

| Failure Scenario | Layer 1 (CronCreate) | Layer 2 (SessionStart) |
|------------------|---------------------|------------------------|
| Stop hook crash (session alive) | Fires after cooldown | N/A |
| Session OOM / SIGKILL | Cannot help | Detects on next start |
| Terminal closed mid-arc | Cannot help | Detects on next start |
| Machine reboot | Cannot help | Detects on next start |
| User cancels arc | Self-deletes (reads flags) | Skips (reads flags) |
| Arc completes normally | Self-deletes (post-arc) | Skips (stop_reason=completed) |

## Monitoring Prompt Template (Layer 1)

Generated by `buildArcMonitoringPrompt(checkpoint)` in `arc-checkpoint-init.md`:

```text
[ARC-MONITOR] Check arc {checkpoint.id}.
Read .claude/arc-phase-loop.local.md.
If missing or user_cancelled=true or stop_reason is "completed" or "user_cancel":
  run CronDelete for this task and stop.
Otherwise read .claude/arc/{checkpoint.id}/checkpoint.json.
If no phase has status "in_progress": run /rune:arc --resume.
If a phase IS in_progress: do nothing (arc is running).
```

## Key Concepts

### Why Layer 1 Is Needed

When a Rune arc pipeline's stop hook fails (due to timeout, error, or other issues), the arc stops mid-execution with no automatic recovery mechanism. The monitoring task provides a best-effort safety net that:

1. Detects unexpected stops (active arc with no in_progress phase)
2. Attempts automatic resume via `/rune:arc --resume`
3. Respects user cancellation (user_cancelled flag)
4. Prevents infinite loops via resume limits

### Why Layer 2 Is Needed

Layer 1 (CronCreate) is session-scoped — it cannot survive session crashes. When the
session itself dies (OOM, terminal closed, machine reboot), the checkpoint persists but
no in-session mechanism can detect it. Layer 2 fills this gap by scanning for orphaned
checkpoints at session startup.

### Key Limitations

1. **Layer 1 session-scoped**: Cannot recover from complete session crashes
2. **Layer 1 between-turns only**: If Claude is stuck mid-response, monitoring cannot fire
3. **Layer 1 no catch-up**: If a scheduled time passes while Claude is busy, it fires once when idle
4. **Layer 1 requires Claude Code >= v2.1.71**: CronCreate/CronDelete/CronList tools
5. **Layer 1 disabled by `CLAUDE_CODE_DISABLE_CRON=1`**: Environment variable disables scheduling
6. **Layer 2 requires user action**: Advises user to resume — does not auto-resume by default

## Configuration

Controlled via `talisman.yml` under `arc.scheduler`:

| Setting | Default | Description |
|---------|---------|-------------|
| `enabled` | `true` | Enable/disable scheduled monitoring |
| `interval_minutes` | `15` | Check interval (5, 10, 15, 30) |
| `auto_renew` | `true` | Auto-renew tasks before 3-day expiry |
| `renewal_threshold_hours` | `12` | Renew when < N hours remain |
| `cleanup_orphans_on_start` | `true` | Clean orphaned tasks on session start |
| `auto_resume.enabled` | `true` | Enable automatic resume on unexpected stop |
| `auto_resume.max_total_resumes` | `10` | Maximum total resume attempts per arc |
| `auto_resume.max_consecutive_failures` | `3` | Stop if this many consecutive failures |
| `auto_resume.cooldown_minutes` | `5` | Wait between resume attempts |
| `auto_resume.notify_on_resume` | `true` | Notify user when auto-resume occurs |

## Resume Tracking

The `resume_tracking` object in the checkpoint tracks:

```javascript
resume_tracking: {
  total_resume_count: 0,      // Total resume attempts across arc lifetime
  resume_history: [],          // Array of { timestamp, trigger, phase }
  last_resume_at: null,        // ISO timestamp of last resume attempt
  consecutive_failures: 0      // Failures since last successful phase completion
}
```

### When to Reset `consecutive_failures`

After a **successful phase completion** following a resume, reset `consecutive_failures` to 0. This distinguishes between:
- Transient failures (network, timeout) → reset on success
- Systemic failures (corrupt checkpoint, missing files) → accumulate

## Security Considerations

1. **Prompt injection risk (Layer 1)**: The monitoring prompt is injected as plain text into CronCreate. If a malicious actor modifies the checkpoint file, they could alter prompt template fields.
   - Mitigation: `buildArcMonitoringPrompt()` validates `checkpoint.id` format (`/^arc-[a-zA-Z0-9_-]+$/`) before interpolation (P1 fix, v1.145.0)
   - Mitigation: State file path is hardcoded (not interpolated from checkpoint)

2. **Session isolation (Layer 1)**: The monitoring task checks `config_dir` and `session_id` to ensure it only acts on arcs from the same session.

3. **Session isolation (Layer 2)**: `session-team-hygiene.sh` verifies `owner_pid` is dead and `config_dir` matches before flagging a checkpoint as resumable. Live sessions' checkpoints are never flagged.

4. **Tool restrictions (Layer 1)**: The monitoring prompt is limited to reading files and invoking `/rune:arc --resume` or `CronDelete`.

## Interaction with Cancel Commands

When `/rune:cancel-arc` is invoked:

1. Sets `user_cancelled=true` in state file and checkpoint
2. Sets `stop_reason="user_cancel"`
3. Deletes the monitoring task via `CronDelete`

The monitoring task checks these flags on every fire and self-deletes if the arc was cancelled or completed.

## Task Renewal (3-Day Expiry)

Scheduled tasks expire after 3 days (Claude Code limitation). For long-running arcs:

1. Before expiry, check `scheduler.expires_at`
2. If < `renewal_threshold_hours` remain, create a new task
3. Delete the old task
4. Update `checkpoint.scheduler` with new task info

## Error Handling

| Error | Recovery |
|-------|----------|
| CronCreate fails | Arc continues without monitoring (non-fatal) |
| CronDelete fails | Log warning, continue (task may already be expired) |
| Checkpoint read fails | Skip this cycle, try again next interval |
| Resume fails | Increment `consecutive_failures`, check limits |
| State file missing | Self-delete via CronDelete (arc was cleaned up) |
